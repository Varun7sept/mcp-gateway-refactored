# JWT Explained Simply

## The Real-World Analogy

Imagine a **concert wristband** that you get at the entrance.

| JWT Concept | Real World |
|-------------|-----------|
| JWT_SECRET | The **special stamp machine** that only the concert organizers have |
| JWT Token | The **wristband** itself |
| Username | Your **name written on the wristband** |
| Expiry | The **date** on the wristband — "Valid until July 15" |
| Signature | The **hologram** that proves it's a real wristband, not a fake |

---

## How a JWT Token is created (when you login)

When you call `POST /api/auth/login {username: "varun", password: "..."}`, the server runs this code (`auth.go:220`):

```go
func (a *Auth) generateToken(username string) (string, error) {
    claims := jwt.MapClaims{
        "sub": username,                        // your name
        "iat": time.Now().Unix(),               // when it was issued
        "exp": time.Now().Add(7 * 24 * time.Hour).Unix(),  // expires in 7 days
    }
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString(a.jwtSecret)       // sign with the secret
}
```

The output is a string like:
```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJzdWIiOiJ2YXJ1biIsImlhdCI6MTc0NTY3ODkwMCwiZXhwIjoxNzQ2MjgzNzAwfQ.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

It has 3 parts separated by dots:

| Part | What it contains | Can you read it? |
|------|-----------------|-----------------|
| `header` | Algorithm info (HS256) | Yes (base64 encoded) |
| `payload` | Your username, issue date, expiry date | Yes (base64 encoded) |
| `signature` | Encrypted stamp using the secret | **No** — only the server can verify this |

## How a JWT Token is checked (middleware)

When you make a request with `Authorization: Bearer <token>`, the server runs (`auth.go:129`):

```go
func (a *Auth) ValidateToken(tokenStr string) (string, error) {
    token, err := jwt.Parse(tokenStr, func(token *jwt.Token) (any, error) {
        // 1. Check if the signature was made with the right algorithm
        if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
            return nil, fmt.Errorf("unexpected signing method")
        }
        // 2. Return the secret so the library can verify the signature
        return a.jwtSecret, nil
    })
    // 3. Check if token is expired
    // 4. Extract username from inside
    username, ok := claims["sub"].(string)
    return username, nil
}
```

**Step by step:**
1. Take the token string
2. Split by dots (header.payload.signature)
3. Re-compute signature: `HS256(header + "." + payload, secret)`
4. Compare with the received signature
5. If they match → token is **valid** and **not tampered**
6. Check `exp` field — if current time > expiry → token is **expired**
7. Extract `sub` (username) — now we know who this is

## Is the JWT_SECRET the same as the JWT Token?

**No.** This is the most common confusion.

| | JWT_SECRET | JWT Token |
|---|---|---|
| What it is | A password only the server knows | A ticket given to each user |
| Who has it | Only the server (set in env variable) | The user who logged in |
| Example | `my-super-secret-key-123` | `eyJhbGciOi...` (long string) |
| Size | Short (you choose) | Long (generated by code) |
| Changes? | Never (stays the same) | **Changes every login** |

---

## Does the same user get the same token every time?

**No.** Every time you login, you get a **different** token. Why?

Because the token contains `iat` (issued at time) and `exp` (expiry time). These change every second. And since the signature is computed over the ENTIRE content (including timestamps), the signature also changes.

Example:

```
Login at 1:00pm → Token A: eyJhbG...abc123
Login at 2:00pm → Token B: eyJhbG...xyz789
```

Both tokens are **valid** and both say "username = varun". But the strings look completely different.

---

## Can one user use another user's token?

If User A's token says `sub: "alice"` and User B tries to use it, the server will see:
- The signature is valid (not tampered)
- But `claims["sub"]` will return `"alice"`, not `"bob"`
- So the server treats User B as if they are Alice

That's why **you keep your token secret** — it's your identity.

## Summary

| Question | Answer |
|----------|--------|
| What is JWT_SECRET? | The master password the server uses to sign tokens |
| What is a JWT Token? | A digital ID card with your name, issue date, and expiry |
| Is the token same every login? | **No** — changes because timestamps change |
| Can User A use User B's token? | **Yes** — the token doesn't check WHO is using it, it only checks if the signature is valid |
| Why set JWT_SECRET in deployment? | So the server can consistently sign and verify tokens, even after restart |
